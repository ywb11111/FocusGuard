# FocusGuard AI 实施规范

> 本文用于约束 AI 在 FocusGuard 项目中的协作方式。任何需要 AI 继续参与开发、重构、排错、写文档或做技术方案时，应优先阅读本文，再阅读 `AGENTS.md`、`Git提交规范.md`、`docs/FocusGuard项目方案.md`、`docs/开发计划.md`、`docs/项目背景与发现.md` 和 `docs/开发进度.md`。

## 1. 用户背景与项目目标

用户是软件工程专业大三学生，方向为 Android 开发，当前技术路线以 Kotlin + Jetpack Compose 为主。用户正在实习，工作中会使用 AI 辅助编码，但个人目标不是只依赖 AI 完成功能，而是借助 AI 提升效率，同时真正掌握 Android 工程能力、系统能力和面试可讲的技术深度。

FocusGuard 的定位不是普通 CRUD 或简单番茄钟，而是一个日常可用、现场可展示、技术点扎实的 Android 作品。

项目核心目标：

- 做成一个真实可用的专注环境助手。
- 能在面试现场展示噪声、光照、手机移动、专注记录和报告。
- 覆盖 SensorManager、AudioRecord、Foreground Service、Room、DataStore、WorkManager、权限适配、Compose 性能优化等技术点。
- 让用户能讲清楚架构设计、数据流、系统限制和工程取舍。

## 2. AI 协作总原则

AI 的职责是帮助用户更快、更稳地推进项目，而不是替用户跳过学习过程。

### 2.1 语言与 Skill 偏好

- 与用户对话默认使用中文。
- 代码注释、说明、文档、计划、总结等内容，除非用户特别要求，默认使用中文。
- 代码标识符、API 名称、库名、命令、路径等保留必要英文。
- 做项目时，如果环境中已安装或可用 `superpower` skill，应优先启动/使用该 skill。
- 如果 `superpower` skill 未安装或不可用，应如实说明，不能假装已经使用。

### 2.2 先理解再实现

每次接到开发任务时，AI 必须先确认当前阶段、相关文档和现有代码结构。不得在不了解项目边界的情况下直接堆代码。

优先阅读顺序：

1. `AGENTS.md`
2. `AI实施规范.md`
3. `Git提交规范.md`
4. `docs/FocusGuard项目方案.md`
5. `docs/开发计划.md`
6. `docs/项目背景与发现.md`
7. `docs/开发进度.md`
8. 相关源码文件

### 2.3 以可解释工程能力为目标

AI 生成的代码必须便于用户学习和面试表达。实现时应保留清晰边界：

- UI 不直接操作 `SensorManager`、`AudioRecord`、Room。
- ViewModel 只管理 UI 状态和调用用例/Repository。
- Repository 协调数据源，不承载复杂 UI 逻辑。
- Analyzer/UseCase 处理评分、环境判断、采样策略等业务规则。
- Service 负责后台监测与通知，不直接控制 UI。

### 2.4 小步提交，避免一次性大爆炸

每次任务应优先完成一个可验证的小闭环。例如：

- 先完成专注计时内存状态，再接 Room。
- 先完成光照传感器实时显示，再接专注期间采样保存。
- 先完成 AudioRecord 相对 dB 输出，再接噪声曲线。
- 先完成前台服务通知基础版，再加暂停/结束动作。

不得一次性生成大量未验证、难以解释、难以维护的代码。

## 3. 用户动手与 AI 生成边界

### 3.1 建议用户亲自动手实现

这些内容是项目的学习价值核心，AI 可以讲解、审查、提示，但不应直接全包：

- `SessionViewModel` 的状态流转和计时状态机。
- Room 的基础 Entity、DAO、Database 调用链。
- `FocusRepository` 从内存演示数据迁移到 Room 的第一版。
- `SensorManager` 的注册、监听、注销。
- `callbackFlow` 如何把传感器回调转成 Flow。
- `AudioRecord` 读取 PCM buffer 和 RMS 计算。
- Foreground Service 的通知渠道、启动和停止流程。
- 高频数据节流、窗口化、减少 Compose 重组的核心策略。

### 3.2 AI 可以生成代码，用户负责理解

这些内容可以更多交给 AI 提效，但用户仍要读懂关键结构：

- Compose 页面布局和组件拆分。
- 重复的 UI 卡片、列表、按钮、空状态。
- Room 查询样板和测试样板。
- Hilt Module 绑定和依赖配置。
- DataStore 设置项模板。
- 图表绘制样式。
- 权限说明文案。
- 技术总结和面试表达草稿。

### 3.3 AI 每次生成核心代码后必须说明

- 本次代码属于“建议用户亲自动手”还是“AI 可代写但用户需理解”。
- 用户应该重点看哪几个文件和哪几段逻辑。
- 这段代码在面试中可以怎么讲。
- 是否需要用户自己复写一遍或做一个小练习。

## 4. 实施流程规范

### 4.1 每次开发前

AI 应先回答这几个问题：

1. 当前要做的是哪个阶段？
2. 本次改动涉及哪些模块？
3. 是否会影响已有架构边界？
4. 是否需要新增依赖、权限或系统能力？
5. 完成后如何验证？

### 4.2 开发中

AI 应遵守：

- 优先复用现有包结构和命名风格。
- 新增类名要能表达职责。
- 数据模型、实体模型、UI 状态模型要分清，不随意混用。
- 高频数据必须考虑节流、采样窗口或局部绘制。
- 涉及权限、后台服务、音频、传感器时必须考虑生命周期释放。
- 涉及隐私的数据，如录音，只保存必要分析结果，不默认保存原始音频。

### 4.3 开发后

AI 应完成：

1. 运行构建或说明无法运行的原因。
2. 更新 `docs/开发进度.md`。
3. 如果阶段、计划、规则或发现发生变化，更新对应文档。
4. 如果关键逻辑变化，补充必要代码注释。
5. 告诉用户本次是否已经形成一个小功能闭环。
6. 如果形成闭环，提醒用户按 `Git提交规范.md` 提交 Git。

## 5. 阶段实施标准

### 阶段 1：基础 App 骨架

完成标准：

- App 能编译运行。
- 有底部导航。
- 有 Today、Session、Reports、Settings 页面。
- 项目分层清晰。
- 已接入或预留 Hilt、Room、DataStore、WorkManager。

### 阶段 2：专注计时与本地记录

完成标准：

- 可以开始、暂停、结束一次专注。
- 专注记录写入 Room。
- Today 页面能显示今日统计。
- Reports 页面能显示历史列表。
- SessionDetail 能显示本次记录详情。

用户应掌握：

- ViewModel 中计时状态如何流动。
- Room 写入和 Flow 查询如何驱动 UI。
- 时间戳和时长如何计算。

### 阶段 3：光照与移动检测

完成标准：

- `LightSensorDataSource` 输出实时光照。
- `MotionSensorDataSource` 输出移动状态。
- 进入/退出页面或服务时能正确注册和释放监听。
- 专注结束后能保存平均光照和移动次数。

### 阶段 4：噪声检测

完成标准：

- 能申请录音权限。
- `AudioRecord` 能读取 PCM buffer。
- 能计算 RMS 和相对 dB。
- UI 能显示实时噪声。
- 专注期间能保存噪声样本。

### 阶段 5：后台服务与通知

完成标准：

- `FocusMonitorService` 可作为前台服务运行。
- 通知渠道正确创建。
- 通知展示当前专注状态。
- 支持暂停/结束操作。
- App 切后台后仍能维持必要监测。

### 阶段 6：报告、优化与面试包装

完成标准：

- 有周报/月报。
- 有趋势图和评分拆解。
- 有性能优化记录。
- 有项目技术总结 Markdown。
- 能讲清楚至少 3 个真实问题和解决方案。

## 6. 文档与注释维护规范

每当项目有更新时，AI 应判断是否需要同步：

- 功能进度变化：更新 `docs/开发进度.md`。
- 阶段计划变化：更新 `docs/开发计划.md`。
- 新发现、踩坑、环境问题：更新 `docs/项目背景与发现.md`。
- 协作规则变化：更新 `AGENTS.md` 或 `AI实施规范.md`。
- Git 流程变化：更新 `Git提交规范.md`。
- 核心代码逻辑变化：补充或调整中文注释。

注释原则：

- 解释“为什么这样做”和“层与层如何协作”。
- 不写机械注释，例如“给变量赋值”“调用函数”。
- 对 ViewModel 状态流、Room 查询、传感器生命周期、AudioRecord 资源释放、前台服务限制等重点位置加注释。

## 7. AI 禁止事项

AI 不应：

- 为了快而绕过架构边界。
- 把所有逻辑塞进 Activity、Composable 或 ViewModel。
- 未说明原因就引入大型第三方库。
- 生成无法编译的“大而全”代码后不验证。
- 在用户需要学习的核心模块只给代码不给解释。
- 把演示数据说成真实采样数据。
- 忽略权限拒绝、传感器缺失、后台限制等真实 Android 问题。
