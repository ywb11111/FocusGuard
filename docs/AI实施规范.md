# FocusGuard AI 实施规范

> 本文用于约束 AI 在 FocusGuard 项目中的协作方式。任何需要 AI 继续参与开发、重构、排错、写文档或做技术方案时，应优先阅读本文，再阅读 `docs/FocusGuard项目方案.md`、`task_plan.md`、`findings.md` 和 `progress.md`。

## 1. 用户背景与项目目标

用户是软件工程专业大三学生，方向为 Android 开发，当前技术路线以 Kotlin + Jetpack Compose 为主。用户正在实习，工作中会使用 AI 辅助编码，但个人目标不是只依赖 AI 完成功能，而是借助 AI 提升效率，同时真正掌握 Android 工程能力、系统能力和面试可讲的技术深度。

用户的第一个项目偏向无障碍/通知栏自动记账方向，已经接触过 Compose、Room、Repository、ViewModel、通知监听、图表等应用层能力。第二个项目 FocusGuard 的定位不是普通 CRUD 或简单番茄钟，而是一个日常可用、现场可展示、技术点扎实的 Android 作品。

FocusGuard 的核心目标是：

- 做成一个真实可用的专注环境助手。
- 能在面试现场展示噪声、光照、手机移动、专注记录和报告。
- 覆盖 SensorManager、AudioRecord、Foreground Service、Room、DataStore、WorkManager、权限适配、Compose 性能优化等技术点。
- 让用户能讲清楚架构设计、数据流、系统限制和工程取舍。

## 2. AI 协作总原则

AI 的职责是帮助用户更快、更稳地推进项目，而不是替用户跳过学习过程。

### 2.1 先理解再实现

每次接到开发任务时，AI 必须先确认当前阶段、相关文档和现有代码结构。不得在不了解项目边界的情况下直接堆代码。

优先阅读顺序：

1. `docs/AI实施规范.md`
2. `docs/FocusGuard项目方案.md`
3. `task_plan.md`
4. `findings.md`
5. `progress.md`
6. 相关源码文件

### 2.2 以可解释工程能力为目标

AI 生成的代码必须便于用户学习和面试表达。实现时应保留清晰的边界：

- UI 不直接操作 `SensorManager`、`AudioRecord`、Room。
- ViewModel 只管理 UI 状态和调用用例/Repository。
- Repository 协调数据源，不承载复杂 UI 逻辑。
- Analyzer/UseCase 处理评分、环境判断、采样策略等业务规则。
- Service 负责后台监测与通知，不直接控制 UI。

### 2.3 小步提交，避免一次性大爆炸

每次任务应优先完成一个可验证的小闭环。例如：

- 先完成专注计时内存状态，再接 Room。
- 先完成光照传感器实时显示，再接专注期间采样保存。
- 先完成 AudioRecord 相对 dB 输出，再接噪声曲线。
- 先完成前台服务通知基础版，再加暂停/结束动作。

不得一次性生成大量未验证、难以解释、难以维护的代码。

### 2.4 保持项目可编译

任何代码改动后，AI 应尽量运行：

```powershell
.\gradlew.bat :app:assembleDebug
```

如果构建失败，必须说明失败原因、已尝试的修复方式和下一步建议。不能假装已验证。

## 3. 用户必须重点掌握的内容

下面这些内容 AI 可以辅助实现，但不能只给最终代码。AI 应同时给用户提供关键解释、调用链和面试讲法。

### 3.1 必须掌握

- `StateFlow`、`SharedFlow`、`callbackFlow` 的区别和适用场景。
- ViewModel 如何暴露 UI state。
- Room 的 Entity、DAO、Database、Repository 分工。
- Navigation Compose 的路由、传参和返回栈。
- Compose 重组原因，以及为什么高频数据不能直接驱动整页重组。
- Android 权限请求与权限拒绝后的降级策略。
- Foreground Service 为什么必须显示通知。
- Android 8+ 后台限制和 Android 13+ 通知权限。

### 3.2 必须能讲清楚

- `SensorManager` 如何注册、监听、注销传感器。
- `AudioRecord` 如何读取 PCM buffer。
- RMS 如何转换为相对噪声值。
- 为什么手机麦克风不能当作专业分贝仪。
- 加速度计如何通过向量模长判断明显移动。
- Repository 为什么隔离数据源。
- 评分规则为什么要可解释，而不是黑盒。
- 高频采样数据如何节流、窗口化、降低 Compose 重组压力。

### 3.3 AI 可以重点辅助

- Compose 页面结构和组件拆分。
- Room DAO 和聚合查询模板。
- 权限说明文案。
- 图表绘制代码。
- 测试样板。
- 技术总结文档。
- 面试表达整理。

## 4. 实施流程规范

### 4.1 每次开发前

AI 应先回答这几个问题：

1. 当前要做的是哪个阶段？
2. 本次改动涉及哪些模块？
3. 是否会影响已有架构边界？
4. 是否需要新增依赖、权限或系统能力？
5. 完成后如何验证？

如果问题明确，可以直接实现；如果需求会影响技术路线，应先给出简短方案再动手。

### 4.2 开发中

AI 应遵守：

- 优先复用现有包结构和命名风格。
- 新增类名要能表达职责，不使用泛泛的 `Manager`、`Helper`，除非确有必要。
- 数据模型、实体模型、UI 状态模型要分清，不随意混用。
- 高频数据必须考虑节流、采样窗口或局部绘制。
- 涉及权限、后台服务、音频、传感器时必须考虑生命周期释放。
- 涉及隐私的数据，如录音，只保存必要的分析结果，不保存原始音频，除非用户明确要求。

### 4.3 开发后

AI 应完成：

1. 运行构建或说明无法运行的原因。
2. 更新 `progress.md`，记录做了什么、验证结果和遗留问题。
3. 如果阶段发生变化，更新 `task_plan.md`。
4. 用简洁语言告诉用户改了哪些关键文件。
5. 标出用户需要亲自理解的技术点。

## 5. 阶段实施标准

### 阶段 1：基础 App 骨架

完成标准：

- App 能编译运行。
- 有底部导航。
- 有 Today、Session、Reports、Settings 页面。
- 项目分层清晰。
- 已接入或预留 Hilt、Room、DataStore、WorkManager。

AI 注意事项：

- 可以用演示数据，但必须明确这是占位实现。
- 不要伪装成真实传感器数据。

### 阶段 2：专注计时与本地记录

完成标准：

- 可以开始、暂停、结束一次专注。
- 专注记录写入 Room。
- Today 页面能显示今日统计。
- Reports 页面能显示历史列表。
- SessionDetail 能显示本次记录详情。

用户应掌握：

- ViewModel 中计时状态如何流动。
- Room 写入和 Flow 查询如何驱动 UI。
- 时间戳和时长如何计算。

### 阶段 3：光照与移动检测

完成标准：

- `LightSensorDataSource` 输出实时光照。
- `MotionSensorDataSource` 输出移动状态。
- 进入/退出页面或服务时能正确注册和释放监听。
- 专注结束后能保存平均光照和移动次数。

用户应掌握：

- `SensorEventListener` 生命周期。
- `callbackFlow` 如何把回调转成 Flow。
- 加速度模长和防抖逻辑。

### 阶段 4：噪声检测

完成标准：

- 能申请录音权限。
- `AudioRecord` 能读取 PCM buffer。
- 能计算 RMS 和相对 dB。
- UI 能显示实时噪声。
- 专注期间能保存噪声样本。

用户应掌握：

- PCM、采样率、buffer size。
- RMS 计算。
- 音频线程和资源释放。
- 隐私说明和权限拒绝降级。

### 阶段 5：后台服务与通知

完成标准：

- `FocusMonitorService` 可作为前台服务运行。
- 通知渠道正确创建。
- 通知展示当前专注状态。
- 支持暂停/结束操作。
- App 切后台后仍能维持必要监测。

用户应掌握：

- 前台服务启动限制。
- 通知权限适配。
- Service 与 Repository/ViewModel 的边界。

### 阶段 6：报告、优化与面试包装

完成标准：

- 有周报/月报。
- 有趋势图和评分拆解。
- 有性能优化记录。
- 有项目技术总结 Markdown。
- 能讲清楚至少 3 个真实问题和解决方案。

用户应掌握：

- Room 聚合查询。
- Compose Canvas 性能策略。
- Macrobenchmark/Baseline Profile 基本使用。
- 如何把项目讲成工程经验，而不是功能堆砌。

## 6. 代码质量规范

### 6.1 命名

- Entity 后缀用于 Room 表模型。
- UiState 后缀用于页面状态。
- Repository 接口放在数据访问边界。
- DataSource 用于传感器、音频、系统 API 等原始数据来源。
- Analyzer 用于可解释规则计算。
- UseCase 用于跨 Repository 或复杂业务流程。

### 6.2 状态管理

- UI 层只收集稳定的 `StateFlow`。
- 高频传感器数据不要直接让整个页面重组。
- 需要实时曲线时，优先使用固定长度窗口。
- 长列表使用 `LazyColumn`，并提供稳定 key。

### 6.3 错误处理

必须考虑：

- 权限拒绝。
- 设备没有光照传感器。
- 麦克风被占用。
- AudioRecord 初始化失败。
- 前台服务启动失败。
- 数据库写入失败。
- App 被系统回收。

错误应转换为可展示的 UI 状态或日志，不应直接崩溃。

### 6.4 权限与隐私

- 不在 App 启动时连续请求权限。
- 权限请求前要有解释。
- 录音只用于计算噪声，不默认保存原始音频。
- 使用情况访问权限必须跳系统设置页，并说明用途。
- 权限拒绝后 App 仍应保留可用的专注计时和本地记录功能。

## 7. 文档与学习输出规范

每完成一个关键模块，AI 应辅助用户沉淀一段技术总结，建议放入 `docs/技术总结.md` 或对应模块文档。

总结格式：

```markdown
## 模块名

### 做了什么

### 为什么这样设计

### 关键代码路径

### 遇到的问题

### 解决方案

### 面试可以怎么讲
```

重点不是写漂亮文案，而是让用户能真正复述。

## 8. AI 禁止事项

AI 不应：

- 为了快而绕过架构边界。
- 把所有逻辑塞进 Activity、Composable 或 ViewModel。
- 未说明原因就引入大型第三方库。
- 生成无法编译的“大而全”代码后不验证。
- 在用户需要学习的核心模块只给代码不给解释。
- 把演示数据说成真实采样数据。
- 忽略权限拒绝、传感器缺失、后台限制等真实 Android 问题。
- 为了界面效果做复杂装饰，牺牲工具型 App 的清晰度。

## 9. AI 每次交付时的回答格式

建议 AI 最终回答包含：

1. 本次完成了什么。
2. 改了哪些关键文件。
3. 如何验证，结果是什么。
4. 用户需要重点看懂的点。
5. 下一步建议。

回答应简洁，不堆无意义术语。
