# FocusGuard Git 提交规范

> 本文规定 FocusGuard 项目的 Git 提交节奏、提交信息格式和提交前检查。每完成一个小功能闭环后，AI 需要提醒用户提交 Git。

## 1. 提交原则

- 小步提交：一个提交只表达一个相对完整的功能、修复或文档更新。
- 可验证：提交前尽量保证项目能编译，优先运行 `.\gradlew.bat :app:assembleDebug`。
- 可回退：不要把多个无关改动混在一个提交里。
- 可说明：提交信息要能看出“做了什么”和“为什么做”。

## 2. 什么时候提交

建议在这些节点提交：

- 完成一个小功能，例如“内存版专注计时状态机”。
- 完成一次数据层接入，例如“专注记录写入 Room”。
- 完成一次页面闭环，例如“Reports 显示真实历史记录”。
- 完成一次文档整理或规范更新。
- 修复一个明确构建错误或运行问题。

不建议等到一天结束才提交一大坨代码。

## 3. 提交信息格式

推荐格式：

```text
类型: 简短描述
```

常用类型：

```text
feat: 新功能
fix: 修复问题
docs: 文档更新
refactor: 重构，不改变行为
test: 测试相关
build: 构建配置或依赖
chore: 杂项维护
```

示例：

```text
feat: 实现专注计时内存状态机
docs: 补充 AI 实施规范和 Git 提交规范
fix: 修复 Kotlin 与 Java JVM target 不一致问题
refactor: 拆分专注页状态展示组件
build: 接入 Room 和 Hilt 依赖
```

## 4. 提交前检查清单

提交前尽量确认：

- 代码能编译。
- 没有无关文件混入。
- 文档和注释已同步更新。
- 演示数据和真实数据的边界说清楚了。
- 如果是核心学习模块，用户已经理解关键逻辑。

建议命令：

```powershell
git status
.\gradlew.bat :app:assembleDebug
git add .
git commit -m "feat: 简短描述"
```

## 5. AI 提醒规则

AI 在每次完成小功能闭环后，应主动提醒：

```text
这次已经形成一个小功能闭环，建议提交一次 Git。
建议提交信息：feat: xxx
```

如果只是中间半成品，不应催促提交功能提交，可以建议等验证通过后再提交。
